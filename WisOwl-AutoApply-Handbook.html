01 · Executive Summary

WisOwl Auto-Apply is a Manifest V3 Chrome Extension that automates job application form filling on company career pages. It runs entirely on the user's device, using their real browser fingerprint and IP address, eliminating bot detection and proxy infrastructure costs.

Key Value Proposition
Infrastructure cost drops from ~$12.50/user/month (cloud automation) to ~$1.75/user/month (on-device). The harder the ATS form, the bigger the advantage.

Core Capabilities

  • Authenticates via WisOwl JWT passed from the web app
  • Fetches matched jobs + pre-generated answers from WisOwl API
  • Opens each career page in a controlled background tab
  • Detects ATS type and selects an appropriate filling strategy
  • Fills all field types: text, select, radio, checkbox, file, date, rich text, custom dropdowns
  • Handles multi-page forms, dynamic conditional fields, and async file uploads
  • Submits the form and captures confirmation text/URL
  • Reports every outcome back to WisOwl (applied / failed / needs_review / skipped)
  • Survives Chrome MV3 service worker restarts via persistent storage

02 · What This Extension Does

User Flow

  1. Auth Handoff — User logs in at app.wisowl.com. The web app calls chrome.runtime.sendMessage(EXT_ID, tokenPayload). The extension stores the JWT in chrome.storage.local.
  2. Session Start — User clicks Start in the popup. The background worker fetches profile, job queue, and a fresh signed resume URL from the WisOwl API.
  3. Job Iteration — For each job:
    • Open career page tab (background)
    • Inject content script
    • Detect ATS → load strategy
    • Detect all form fields
    • Map profile data to fields (deterministic → QA bank → LLM fallback)
    • Fill fields with anti-detection delays and event chains
    • Handle multi-page navigation
    • Click Submit and capture confirmation
    • Log result to WisOwl API
    • Close tab, wait 2–5s, proceed to next job
  4. Session End — When queue completes, show summary popup with applied / failed / needs_review counts.

Anti-Detection Behaviors

RuleImplementation
Random delay between fields200–600ms per field
Character-by-character typing8–25ms/char for text > 100 chars
Random delay between jobs2,000–5,000ms
Focus before fill, blur afterFull synthetic event chain
Never batch-fill synchronouslyStrict async/await with delays
Visual order fillingTop-to-bottom, left-to-right
Scroll to field before fillingscrollIntoView({ block: 'center' })
Skip hidden fieldsdisplay:none, visibility:hidden, aria-hidden
Hard Rule
Never submit a form that has any required field left empty due to a mapping failure. If a required field cannot be filled, the job is logged as needs_review and skipped.

MV3 Survival Strategy

Chrome MV3 service workers terminate after ~30s of inactivity. The extension handles this by:

  1. Storing all session state in chrome.storage.local
  2. Creating a keepalive alarm every 24s (chrome.alarms.create('keepalive', { periodInMinutes: 0.4 }))
  3. On worker restart, reading the is_running flag from storage
  4. If is_running === true but the process was killed, resuming from the last known job index

04 · Blocking Dependencies

The extension cannot function without these inputs. They must be built and deployed before the extension ships.

A. Backend API Endpoints REQUIRED

MethodEndpointResponse Shape
GET/extension/profileUserProfile (see §06)
GET/extension/job-queue?date=YYYY-MM-DDJobQueue with pre-generated content
GET/extension/resume-url{ signed_url, expires_at }
POST/extension/llm-assist{ answer, confidence }
POST/extension/result201 Accepted
POST/extension/session-end201 Accepted

B. Pre-Generation Service REQUIRED

Must run before each daily session (nightly batch recommended). For every high/medium match job, generates:

  • cover_letter: 150–250 words, role-specific, first person
  • qa_bank: 8–10 regex patterns covering common form questions (salary, notice period, motivation, strengths, etc.)

Uses Claude Sonnet for quality. System prompt must forbid AI vocabulary and enforce first-person voice.

C. JWT Auth System REQUIRED

The WisOwl web app must generate a JWT and pass it to the extension via chrome.runtime.sendMessage(EXTENSION_ID, payload). The payload must include:

{ "token": "jwt_string", "expires_at": "ISO8601", "user_id": "string", "extension_id": "string" }

D. Resume CDN with Signed URLs REQUIRED

User resume PDFs must be hosted on a CDN (S3, R2, etc.) with signed URLs. TTL must be ≥ 60 minutes. The backend returns a fresh signed URL at session start.

E. Extension ID Integration REQUIRED

The WisOwl web app must hardcode the extension's Chrome Extension ID to send the auth token. This ID is obtained from chrome://extensions after loading the unpacked extension, or from the Chrome Web Store developer dashboard after publishing.

Critical Integration Point
If the web app sends messages to the wrong Extension ID, auth fails silently. The popup will show "Connect to WisOwl" forever. The Extension ID must be embedded in the web app at build time.

F. Chrome Web Store Assets OPTIONAL FOR DEV

  • Icons: 16×16, 48×48, 128×128 PNG files in icons/
  • Privacy policy (required for <all_urls> permission)
  • Justification for broad host permission: "The extension needs to inject content scripts on arbitrary company career pages to fill application forms."

05 · Developer Setup Guide

Prerequisites

  • Node.js 18+ and npm
  • Google Chrome (desktop only — MV3 extensions do not run on Android)
  • Git

Step 1: Clone & Install

git clone https://github.com/MadhavShankar/autoapplychromeextension.git
cd autoapplychromeextension
npm install

Step 2: Configure

Edit lib/config.ts:

export const config = {
  apiBaseUrl: 'https://api.wisowl.com/v1',   // ← your API domain
  dailyCap: 25,                               // ← match backend
  // ... other settings
};

Edit public/manifest.json:

"externally_connectable": {
  "matches": ["https://app.wisowl.com/*"]   // ← your web app origin
}

Step 3: Build

npm run build

Output goes to dist/ with this structure:

dist/
├── manifest.json
├── background.js              (ES module)
├── content/
│   └── form-agent.js          (IIFE bundle)
├── popup/
│   ├── index.html
│   ├── popup.css
│   └── popup.js
└── icons/
    ├── icon16.png
    ├── icon48.png
    └── icon128.png

Step 4: Load in Chrome

  1. Open chrome://extensions
  2. Enable Developer mode (toggle top-right)
  3. Click Load unpacked
  4. Select the dist/ folder
  5. Copy the Extension ID from the extension card

Step 5: Integrate with Web App

Embed the Extension ID in your web app frontend:

const EXTENSION_ID = 'abcdefghijklmnopabcdefghijklmnop';

function handoffToken(jwt, userId, expiresAt) {
  chrome.runtime.sendMessage(EXTENSION_ID, {
    token: jwt,
    user_id: userId,
    expires_at: expiresAt,
    extension_id: EXTENSION_ID
  });
}

Step 6: Run Tests

npm test           # run once
npm run test:watch # watch mode

Development Mode

npm run dev

Starts Vite dev server with HMR for popup UI. Background worker changes require manual extension reload in chrome://extensions.

POST /extension/llm-assist

// Request
{ "job_id": "string", "field_label": "string", "field_type": "text | select | radio | checkbox", "options": ["string"], "context": "string", "resume_snippet": "string" }

// Response
{ "answer": "string", "confidence": "high | medium | low" }

If confidence is low, the extension marks the field as needs_review and does not fill it.

POST /extension/result

{ "record": ApplicationRecord }

Called immediately after each job attempt. No batching. If the call fails, the result is queued in chrome.storage.local and retried on the next session start.

11 · Troubleshooting

Popup shows "Connect to WisOwl" forever

  • Check that the web app origin matches externally_connectable.matches in manifest
  • Verify the Extension ID in the web app matches the installed extension
  • Open background service worker console (chrome://extensions → click "service worker" link) and check for [Auth] logs

Session starts but no tabs open

  • Check background console for API errors (401 = expired JWT, 5xx = backend down)
  • Verify apiBaseUrl in lib/config.ts is correct and reachable
  • Ensure job queue is not empty (all jobs may have already_applied: true)

Fields not detected on a specific site

  • Open DevTools on the career page → Console
  • Check if fields are inside cross-origin iFrames (P2 limitation)
  • Check if fields are rendered by React/Angular with delayed hydration (increase maxDomWaitMs in config)
  • File an issue with the URL and a DOM snapshot

Resume upload fails

  • Check background console for FILE_FETCH_FAIL — signed URL may be expired
  • Verify resume PDF is accessible from the signed URL in a normal browser tab
  • Check if the form uses drag-and-drop only (Scenario C) — detection may need tuning

Service worker restarts mid-session

  • This is expected MV3 behavior. The extension is designed to resume automatically.
  • Check chrome.storage.local for wisowl:is_running flag
  • Verify keepalive alarm is firing (check chrome.alarms in background console)
