WisOwl Auto-Apply · Chrome Extension PRD

Phase 1 · Final Chrome Extension · Manifest V3 TypeScript · Vanilla DOM v2.0 · May 2025 Author: Madhav Shankar · WisOwl

01 · Product Summary

A Chrome Extension that automatically fills and submits job applications on company career pages, running on the user's own device. It reads matched jobs from WisOwl's backend (high + medium scores), navigates to each career page, fills every form field using the user's profile and pre-generated answers, and reports outcomes back to WisOwl.

Why on-device, not cloud
The extension runs from the user's real IP with their real browser fingerprint. Bot detection, CAPTCHAs, and proxy costs are eliminated. Infrastructure cost is ~$1.75/user/month vs ~$12.50/user/month for a cloud approach. The harder the form, the bigger the advantage.

What the extension does

  • Authenticates via WisOwl JWT (passed from the WisOwl web app)
  • Fetches daily matched job list and user profile from WisOwl API
  • Opens each career page in a controlled tab
  • Detects ATS type and selects appropriate filling strategy
  • Fills all field types: text, select, radio, checkbox, file, date, rich text, custom dropdowns
  • Handles multi-page forms, iFrame-embedded forms, and async file uploads
  • Submits the form and captures the confirmation
  • Logs every outcome to WisOwl (applied / failed / needs_review / skipped)
  • Shows live progress in the extension popup

03 · Tech Stack

Extension (Client)
  • TypeScript (strict mode)
  • Chrome Extension Manifest V3
  • Vanilla DOM — no React/Vue (keep bundle < 200KB)
  • Vite for bundling
  • chrome.storage.local for all state
Form Automation
  • Native Chrome DOM APIs (content scripts)
  • No Playwright, no Selenium, no headless browser
  • DataTransfer API for file injection
  • MutationObserver for dynamic field detection
LLM (Ambiguous Fields Only)
  • Claude Haiku via WisOwl backend proxy
  • Called from background service worker only
  • ~10% of fields need LLM; rest are deterministic
  • API key never touches the extension
WisOwl Backend (Pre-existing)
  • REST API with JWT auth
  • Pre-generation service must be built before extension ships
  • Resume PDF served via signed CDN URL
  • Result logging endpoint

MV3 Service Worker Constraint
Chrome MV3 service workers terminate after 30s of inactivity. All session state must live in chrome.storage.local. Use chrome.alarms as a keepalive ping every 25s during active sessions. On service worker restart, read state from storage and resume cleanly.

If already_applied: true, skip immediately — no tab, no navigation. The backend is the source of truth for deduplication; the extension just respects the flag.

Input C — Auth Token

Passed from WisOwl web app to extension via chrome.runtime.sendMessage. Stored in chrome.storage.local. Never in memory only.

{
  "token":        "string (JWT)",
  "expires_at":   "ISO8601",
  "user_id":      "string",
  "extension_id": "string"     // WisOwl web app uses this to route the message
}
Token Handoff Flow
User logs in at app.wisowl.com. Web app calls chrome.runtime.sendMessage(EXTENSION_ID, tokenPayload). Background service worker receives via chrome.runtime.onMessageExternal. Popup updates to Ready state. The extension ID must be embedded in the WisOwl web app at build time.

07 · WisOwl API Contracts

Base URL & Auth
All endpoints relative to WISOWL_API_BASE (configured in extension, default: https://api.wisowl.com/v1). All requests: Authorization: Bearer <token>. On 401, extension stops session and shows "Session expired — please log in again."

GET/extension/profile

Returns full user profile (Input A shape). Called once at session start. Response cached in storage for the session.

GET/extension/job-queue?date=YYYY-MM-DD

Returns today's job list (Input B shape). Jobs ordered: high match first, then medium. Backend applies deduplication before returning.

GET/extension/resume-url

Returns a fresh signed URL for the resume PDF. Called at session start (not per-job). TTL must be at least 60 minutes to cover a full session.

Response: { "signed_url": "string", "expires_at": "ISO8601" }

POST/extension/llm-assist

Called by background worker when content script sends an unmappable field. Backend calls Claude Haiku and returns the answer. Keeps API key server-side.

Body:
{
  "job_id":          "string",
  "field_label":     "string",       // exact label from form DOM
  "field_type":     "text | textarea | select | radio | checkbox",
  "options":        ["string"],      // available options, if applicable
  "context":         "string",       // surrounding section heading for context
  "resume_snippet":  "string"        // first 500 chars of resume text
}

Response:
{
  "answer":      "string",
  "confidence":  "high | medium | low"
}
Rule
If confidence is low, extension marks field as needs_review and does NOT fill it. This prevents a wrong answer being submitted.

POST/extension/result

Called immediately after each job attempt. Do not batch. Accepts ApplicationRecord body.

POST/extension/session-end

Body: { "session_id": "string", "summary": SessionState }

08 · Pre-Generation Service Contract

Backend Must Build — Blocking Dependency
This service runs on WisOwl's server before each daily session. It generates cover letters and Q&A banks per job. The extension relies entirely on this output — it does not generate content itself.

When it runs

Triggered nightly (e.g. 2am IST) for all users with pending high/medium matches. Output attached to each job in the job-queue API response.

Input per job

{
  "user_profile":    UserProfile,   // full profile object
  "job_title":       "string",
  "company":         "string",
  "job_description": "string"
}

Output per job (attached to job in queue)

{
  "cover_letter": "string",      // 150-250 words, role-specific, first person
  "qa_bank": [
    {
      "question_pattern": "why.*this role|motivation|interest in", // regex
      "answer": "string"          // 2-3 sentence answer
    },
    {
      "question_pattern": "strength|what.*good at",
      "answer": "string"
    },
    {
      "question_pattern": "salary|compensation|ctc|package",
      "answer": "string"          // uses expected_ctc from profile
    },
    {
      "question_pattern": "notice|join|available|start date",
      "answer": "string"          // uses notice_period from profile
    }
    // minimum 8-10 patterns covering common Indian job form questions
  ]
}
LLM Prompt Guidance (for the backend team)
Use Claude Sonnet for pre-generation (quality matters here, not speed). System prompt must instruct: no AI vocabulary, first-person voice, specific to the job title and company, reference 1-2 actual skills from the user's profile. Max 250 words for cover letters. Output JSON only.

11 · Components

background/worker.ts — Orchestrator

  • Receives auth token via chrome.runtime.onMessageExternal from app.wisowl.com
  • On session start: calls profile, job-queue, and resume-url APIs. Stores all to storage.
  • Iterates job queue: opens tab → injects content scripts → waits for result message → closes tab → logs result → delays 2–5s → next job
  • Sends SESSION_UPDATE to popup on every state change
  • On LLM_ASSIST_REQUEST from content script: calls POST /extension/llm-assist, returns result
  • Keepalive: chrome.alarms.create('keepalive', { periodInMinutes: 0.4 })
  • On alarm: check storage for active session, re-register alarm, continue session if interrupted

content/form-agent.ts — Entry Point (Per Tab)

  • Receives job data and profile via chrome.runtime.sendMessage after injection
  • Waits for DOM ready (max 5s)
  • Calls ATS detector → selects filling strategy
  • Calls field-detector to build field inventory
  • For parse-from-resume portals: calls resume-uploader first, then waits 3s for autofill, then scans remaining unfilled fields
  • Calls field-mapper to map each field
  • Calls field-filler to execute DOM interactions
  • After each field: waits for MutationObserver (dynamic-watcher) to settle before proceeding
  • Calls submit-handler when all fields on current page are filled
  • For multi-page forms: after submit-handler clicks Next, waits for new page, re-runs field-detector
  • Returns final result to service worker

content/field-detector.ts

Reads DOM only. Returns array of DetectedField objects. No filling logic.

// DetectedField shape
{
  "selector":     "string",       // unique CSS selector to locate element
  "label":        "string",       // normalised label text (lowercase, trimmed)
  "type":         "text | email | tel | number | textarea | select | radio | checkbox | file | date | richtext | custom-dropdown | unknown",

Detection order per field: aria-label → htmlFor label → placeholder → name attribute → surrounding text node → parent heading text.

12 · ATS Detection & Strategies

The ATS detector runs first, before form detection. It identifies the ATS from URL patterns and DOM signals, then loads the appropriate strategy module. Strategies override specific behaviours (e.g. iFrame handling, field selector heuristics) while the generic filling logic handles everything else.

ATSDetection SignalKnown QuirksMarket Presence (India)
Greenhouse boards.greenhouse.io in URL Standard HTML form. Very consistent. Easiest to fill. Funded Indian startups, tech cos
Lever jobs.lever.co in URL React-controlled inputs. Needs synthetic event override. Tech companies, SaaS
Workday myworkdayjobs.com or wd3.myworkday in URL iFrame embedded. Multi-step. Slow to load. Login often required (P2). MNCs operating in India
Taleo (Oracle) taleo.net in URL or taleo in page scripts iFrame embedded. Old DOM structure. Often requires account (P2). Large Indian corporates, PSUs
Zoho Recruit zohorecruit.com or zoho.com/recruit in URL Standard form. Consistent field naming. WisOwl's ICP. Mid-size staffing agencies — primary ICP
Darwinbox darwinbox.com in URL or meta tag React SPA. Custom dropdown components. Dynamic sections. Indian enterprise, D2C brands
Keka keka.com/careers or kekaats in URL Angular. Custom file upload component. Indian mid-market HR
Freshteam freshteam.com in URL Standard HTML, simple form. Easiest Indian ATS. SMEs, startups
SAP SuccessFactors successfactors.com in URL Heavy iFrame, slow, often requires account. Log as needs_review in P1. Enterprise, PSUs
Generic / Custom No pattern matched Use generic strategy. Best-effort fill. Startups, SMEs with custom career pages
Strategy Pattern
Each ATS strategy file exports an object with overridable hooks: beforeFill(), detectFields(), afterFieldFill(field), detectSubmitButton(), detectConfirmation(). The generic strategy provides defaults. ATS-specific strategies override only what differs.

13 · Application Flow

1 · Auth Handoff
User logs in at app.wisowl.com. Web app calls chrome.runtime.sendMessage(EXT_ID, tokenPayload). Service worker stores token to storage. Popup shows Ready state.
2 · Session Initialisation
User clicks Start. Service worker calls: GET /profile → GET /job-queue → GET /resume-url. Stores all to storage. Filters already_applied: true. Enforces daily_cap. Sorts: high match first. Registers keepalive alarm.
3 · Tab Open
chrome.tabs.create({ url: job.apply_url, active: false }). Stores active_tab_id. Waits for chrome.tabs.onUpdated with status: 'complete'. Timeout: 30s. On timeout → log TIMEOUT_PAGE, skip job.
4 · Content Script Injection
chrome.scripting.executeScript({ target: { tabId, allFrames: true }, files: ['content/form-agent.js'] }). Immediately sends job data + profile via chrome.tabs.sendMessage.
5 · ATS Detection
Checks URL pattern and DOM signals. Selects ATS strategy module. Falls back to generic if no match.
6 · Parse-from-Resume Check
Before general field filling: check if the page has an upload-first pattern (resume field is at the top, other fields are empty or hidden). If yes → upload resume first → wait 3s for autofill → proceed to fill remaining fields only.
7 · CAPTCHA Pre-Check
Before filling anything, check if a CAPTCHA is already rendered on the page (reCAPTCHA iframe, hCaptcha widget). If yes → enter AWAITING_USER state → surface tab → notify popup → wait for user to solve → resume when user clicks "I've solved it" in popup.
8 · Field Detection & Mapping
field-detector scans DOM for all form fields. field-mapper runs deterministic matching first, then pattern matching against qa_bank, then LLM fallback. Fields with low LLM confidence are flagged needs_review but filling continues on other fields.
9 · Field Filling Loop
For each field: fill using type-appropriate method (see Section 14). After each fill: wait 200–600ms (randomised). After each fill: MutationObserver checks for new fields revealed by the interaction (dynamic fields). If new fields appear, add to queue and continue. Maximum 60 fields per page before logging needs_review.
10 · Multi-Page Navigation
After all visible fields on current page are filled: look for a Next/Continue/Save & Continue button (not the final Submit). Click it. Wait for page content change (URL change or DOM mutation). Re-run field-detector. Repeat until no Next button — only Submit remains.
11 · Submit
submit-handler locates and clicks the final Submit button. Waits up to 10s for page change or success indicator. If a CAPTCHA appears post-submit: enter AWAITING_USER state. If validation errors appear: attempt one auto-fix pass, retry submit once.
12 · Confirmation Capture
Check for: URL change to /thank-you, /confirmation, /success. Or text containing "application received", "thank you", "successfully submitted", "we'll be in touch". Log confirmation text/URL. If neither detected within 10s: log SUBMIT_UNCONFIRMED (needs_review).
13 · Result Logging & Tab Cleanup
Send result to service worker. Service worker posts to POST /extension/result. Closes tab. Updates session state. Waits 2–5s (randomised). Advances to next job.

15 · Resume Upload — All Scenarios

Developer Note
Resume upload is the most complex part of the extension. All four scenarios below must be implemented. The fallback chain is: Scenario A → B → C → D → fail with FILE_UPLOAD_FAIL.

Scenario A · Native input[type=file]

Most common case. Extension fetches PDF in background worker, passes ArrayBuffer to content script, injects via DataTransfer.

// Step 1: Background worker fetches PDF (not content script — avoids CSP block)
// In background/worker.ts:
chrome.runtime.onMessage.addListener(async (msg, sender, sendResponse) => {
  if (msg.type === 'GET_RESUME_FILE') {
    const res = await fetch(signedResumeUrl);
    const buffer = await res.arrayBuffer();
    sendResponse({ buffer, filename, mimeType: 'application/pdf' });
  }
  return true; // keep channel open for async response
});

// Step 2: Content script receives ArrayBuffer, injects into file input
// In content/resume-uploader.ts:
const fileData = await chrome.runtime.sendMessage({ type: 'GET_RESUME_FILE' });
const blob = new Blob([fileData.buffer], { type: 'application/pdf' });
const file = new File([blob], fileData.filename, { type: 'application/pdf' });
const dt = new DataTransfer();
dt.items.add(file);
fileInput.files = dt.files;
fileInput.dispatchEvent(new Event('change', { bubbles: true }));

Scenario B · React / Angular Controlled Input

Setting fileInput.files directly bypasses React's internal state. Must override the native property setter first.

// Override React's property descriptor before setting files
const nativeSetter = Object.getOwnPropertyDescriptor(
  window.HTMLInputElement.prototype, 'files'
).set;

nativeSetter.call(fileInput, dt.files);
fileInput.dispatchEvent(new Event('change', { bubbles: true }));

// Detection signal: React/Angular forms typically have __reactFiber or ng attributes on the input
const isReact    = !!Object.keys(fileInput).find(k => k.startsWith('__reactFiber'));
const isAngular  = fileInput.hasAttribute('ng-model') || fileInput.hasAttribute('[(ngModel)]');

Scenario C · Drag-and-Drop Zone

No input[type=file] present. Form uses a custom div with drop event listeners.

// Detect: look for file input absence + dropzone indicators
const hasFileInput = !!document.querySelector('input[type="file"]');
const dropZone = document.querySelector(
  '[class*="dropzone"],[class*="drop-zone"],[class*="upload-area"],[data-dropzone],[aria-label*="upload"]'
);

if (!hasFileInput && dropZone) {
  // Simulate drag sequence
  dropZone.dispatchEvent(new DragEvent('dragenter', { bubbles: true }));
  dropZone.dispatchEvent(new DragEvent('dragover',  { bubbles: true, cancelable: true }));
  dropZone.dispatchEvent(new DragEvent('drop', {
    bubbles: true, cancelable: true,
    dataTransfer: dt   // DataTransfer object with the file
  }));
  await delay(500);
}

Scenario D · Async Pre-Submit Upload

Form uploads file immediately on selection (before Submit is clicked) — sends to S3 via XHR. Extension must wait for this background upload to complete before proceeding to fill other fields or clicking Submit.

// After injecting file, wait for upload indicators to resolve
const uploadComplete = await waitForCondition(() => {
  const uploading = document.querySelector(
    '[class*="uploading"],[class*="progress"],[aria-label*="uploading"],[class*="spinner"]'
  );
  const done = document.querySelector(
    '[class*="upload-success"],[class*="file-name"],[class*="uploaded"],[aria-label*="uploaded"]'
  );
  return !uploading && !!done;
}, { timeout: 30000, interval: 500 });

if (!uploadComplete) return { status: 'failed', reason: 'FILE_UPLOAD_TIMEOUT' };

Shared Pre-Upload Checks (run before all scenarios)

// 1. File type check — skip upload if PDF not accepted
const accepts = fileInput?.getAttribute('accept') || '';
if (accepts && !accepts.includes('pdf') && !accepts.includes('*')) {
  return { status: 'failed', reason: 'FILE_TYPE_REJECTED' };
}

// 2. File size check
const maxBytes = getMaxFileSizeFromDom(); // parse from label text like "Max 2MB"
if (maxBytes && profile.resume.size_bytes > maxBytes) {
  return { status: 'failed', reason: 'FILE_SIZE_REJECTED' };
}

16 · Custom Dropdown Handling

Many modern career pages use custom searchable dropdowns (react-select, Select2, Chosen, Downshift) instead of native <select>. These look like an input but behave as a typeahead.

// Detection: custom dropdown has no <option> children but has a role="combobox" or class pattern
const isCustomDropdown =
  element.getAttribute('role') === 'combobox' ||
  element.classList.contains('select__input') ||     // react-select
  element.classList.contains('select2-search__field') || // Select2
  element.closest('[class*="select-container"]');

if (isCustomDropdown) {
  // 1. Click the container to open the dropdown
  element.closest('[class*="control"],[class*="container"]')?.click();
  await delay(300);

  // 2. Type the value to filter options
  element.value = mappedValue;
  element.dispatchEvent(new Event('input', { bubbles: true }));
  await delay(400);

  // 3. Find the first matching option in the dropdown menu and click it
  const menu = document.querySelector('[class*="menu"],[class*="dropdown"],[role="listbox"]');
  const option = Array.from(menu?.querySelectorAll('[class*="option"],[role="option"]') || [])
    .find(el => normalize(el.textContent) === normalize(mappedValue));
  option?.click();
}

Many forms reveal new fields based on earlier answers (e.g. selecting "Yes" to "Do you have a visa?" reveals a new field "Visa Type"). The extension must detect and handle these.

// After every field interaction, watch for DOM changes

  const newFields = detectNewFields(mutations);

19 · Anti-Detection Rules

Why This Matters Even on Device
Although the extension runs from the user's real IP, some enterprise ATSs (Workday, Taleo) have JavaScript-level bot detection that checks interaction timing and patterns. These rules ensure the extension's behaviour is indistinguishable from a fast typist.
RuleImplementation
Random delay between fields 200–600ms per field. Use Math.random() * 400 + 200.
Character-by-character typing for long text Any textarea or text input with >100 chars: type at 8–25ms/char.
Random delay between jobs 2,000–5,000ms between closing one tab and opening the next.
Focus before fill, blur after fill Always call .focus() before setting value, .blur() after.
Never batch-fill all fields synchronously Always use async/await with delays between each field.
Tab ordering Fill fields in visual top-to-bottom, left-to-right order matching natural reading flow.
Scroll to field before filling element.scrollIntoView({ behavior: 'smooth', block: 'center' })
Do not fill hidden fields Skip any field with display:none, visibility:hidden, or aria-hidden:true.

20 · State Machine

IDLE
Auth valid. Profile + queue loaded. Waiting for user to click Start.
RUNNING
Active session. Tab is open. Content script is filling a form.
PAUSED
User clicked Pause. Current job completed. Next job not yet started.
AWAITING_USER
CAPTCHA detected or low-confidence field. Tab surfaced. Waiting for user.
ERROR
Fatal session error (e.g. API auth failure). User must restart.
COMPLETED
All jobs in queue processed. Summary screen shown. Session end posted.
// Transitions
IDLE           → RUNNING         : user clicks Start, queue loaded
RUNNING        → PAUSED          : user clicks Pause (after current job completes)
RUNNING        → AWAITING_USER   : CAPTCHA detected | low-confidence required field
RUNNING        → ERROR           : auth expired | unrecoverable API failure
RUNNING        → COMPLETED       : all jobs processed
PAUSED         → RUNNING         : user clicks Resume
AWAITING_USER  → RUNNING         : user resolves CAPTCHA / confirms field / clicks Continue
ERROR          → IDLE            : user dismisses error and restarts
COMPLETED      → IDLE            : user starts a new session

Non-Negotiable Rule
Never submit a form that has any required field left empty due to a mapping failure. Submitting a blank required field damages the user's application. If a required field cannot be filled, the job must be logged as needs_review and skipped. Optional fields may be left blank.

22 · Security Rules

RuleImplementation
JWT token storage Store only in chrome.storage.local. Chrome encrypts this storage on-device. Never log the token.
Resume PDF access Fetched via short-TTL signed URL (≥60 min TTL). URL fetched fresh at session start. Expired URLs fail gracefully.
LLM API key Never present in extension bundle. All LLM calls go through WisOwl backend proxy. Extension only sends field metadata.
No third-party credential storage Extension never stores usernames or passwords for career portals.
externally_connectable restriction Only https://app.wisowl.com/* can send messages to the extension. All other sources are rejected.
Content script isolation Content scripts run in an isolated world. They cannot access page JavaScript variables. Career pages cannot access extension storage or credentials.
User profile data exposure Profile data is passed to content scripts only as needed per job. Full profile is not embedded in the extension bundle.
Session single-device lock Backend must set a session lock on POST /extension/session-end and reject job-queue requests if a session is already active for that user_id.

24 · Out of Scope — Phase 1

  • Scheduled / automatic runs — Extension requires open browser. Scheduling is a Phase 2 desktop app feature.
  • Login-required portals — Workday, Taleo with mandatory accounts. P2 flow: user logs in, extension resumes.
  • Cross-origin iFrame scripting — Chrome blocks this. P2 workaround TBD.
  • Invisible CAPTCHAs (reCAPTCHA v3, Turnstile) — Score-based, undetectable pre-submit. P2 strategy needed.
  • LinkedIn Easy Apply — Separate module with its own DOM and ToS considerations.
  • Naukri / Indeed portal flows — Portal-specific modules with API-first path in P2.
  • Application withdrawal — Cannot undo submitted applications.
  • Video intro fields — Skip silently. Log as needs_review if required.
  • SAP SuccessFactors — Requires account creation. Log as needs_review.
  • Application history dashboard — WisOwl web app responsibility. Extension only logs data.
  • Cover letter / Q&A generation — Done by WisOwl pre-generation service. Extension only pastes output.
  • Mobile (Android Chrome) — Chrome extensions are desktop-only.
